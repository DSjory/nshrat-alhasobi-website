# Codebase Documentation — nshrat-alhasobi-website

Purpose
- Provide a single, developer-friendly description of the project idea, architecture, data model, major flows, current issues, and onboarding notes so another engineer or an automation agent can add features reliably.

**Project Idea**
- **Description**: A bilingual (Arabic + English) static site that publishes a curated weekly newsletter called "نشرة الحاسوبي". Public visitors can view newsletter issues, read sections (news, articles, podcast, translations, illumination, etc.), and join the team via a recruitment form. Editors and admins manage newsletters in a small embedded CMS under `/admin_cms`.
- **Primary users**: public readers (newsletter consumers), contributors (join form), and admins/editors (via admin CMS).

**Tech Stack**
- **Frontend**: Vanilla HTML/CSS + modular JavaScript (ES modules). Build tooling: `vite`.
- **Backend-as-a-Service**: Supabase (Postgres DB, Storage bucket `newsletter-media`, Auth, Realtime). DB schema is in [supabase/schema.sql](supabase/schema.sql).
- **Hosting**: Static site built into `dist` and configured for Vercel ([vercel.json](vercel.json)).
- **Node scripts**: `scripts/copy-static.js` copies admin and static assets into `dist` after the Vite build.
- **Dependencies**: See [package.json](package.json) — `vite` (dev), `@supabase/supabase-js` (runtime).

**High-level Architecture**
- Static site generated by Vite into `dist`. The `build` script runs `vite build` then `node scripts/copy-static.js` to include admin assets.
- Public pages query Supabase directly from the browser (public/anon key) for published content.
- Admin CMS uses Supabase Auth (email/password) and uses an env file at `admin_cms/env.json` in current code to initialize the admin client.
- Media (images/audio) are stored in Supabase Storage bucket `newsletter-media` and served via Supabase public URL endpoints.
- Supabase schema includes strong audit logging and Row-Level Security (RLS) policies.

**Data Model (summary)** — See [supabase/schema.sql](supabase/schema.sql)
- `categories` — newsletter categories.
- `section_types` — canonical section types (illumination, inspiring, news, articles, podcast, translation) with optional header image support.
- `newsletters` — header-level newsletter metadata:
  - Core: `title_ar`, `title_en`, `edition_number`, `status` (draft/published/archived), `cover_image_url`
  - New: `reading_time` (estimated read duration), `welcome_message` (greeting text), `has_translation` (boolean flag), `translated_content` (full English translation)
- `newsletter_sections` — association of a newsletter to several section types (order, visibility, **new:** header image support):
  - `header_image_url` — optional banner image for the entire section
  - `header_image_alt_ar` — alt text for accessibility
- Section payload tables (content):
  - `section_illumination`, `section_inspiring` (single-row per section, includes header_image_url for visual sections)
  - `section_news_items` (multi-row news items, each with optional thumbnail)
  - `section_article_items` (multi-row article items)
  - `section_podcast` (audio + metadata)
  - `section_translation` (legacy translation content)
- `join_requests` — recruitment form submissions (public insert allowed by RLS).
- `audit_logs` — append-only audit trail recorded by triggers on domain tables.

**Key Flows (detailed, file references included)**
- **Public list & detail rendering**

  - Single issue page: [single-episode.html](single-episode.html) uses `js/newsletter-render.js`:

    - **Hides** the main `cover_image_url` (removed from single-episode view)
    - **Renders** `welcome_message` (greeting at top) and `reading_time` (metadata)

    - Otherwise: renders all visible sections in order with their content
    - Each section may have a `header_image_url` banner (new feature) rendered before section content

- **Join / recruitment form**
  - Public page: [join.html](join.html) with `js/join-form.js`.
